## Summary

Owen Williams, a design manager at Stripe, shares how he built Protodash—an internal AI prototyping tool that turns Stripe's design system (Sail) into clickable, production-quality prototypes in minutes. What started as a bundle of Cursor rules and an MCP server for Sail evolved into a full browser-based platform used equally by PMs and designers. The episode covers how the tool eliminated "blurple slop" from generic AI prototypers, why PMs became the surprise power users, and Owen's workflow rules for working with AI effectively.

## Key Highlights

- Protodash started as Cursor rules + Sail MCP server integration; now a full web-based prototyping platform
- Core Cursor rule: if user pastes a Figma link, check the Sail MCP server before writing code—don't improvise the design system
- Generic AI tools (e.g., v0) produce "blurple slop"—uncanny-valley Tailwind prototypes that don't match Stripe's design system
- PMs now use Protodash as much as designers, shifting conversations from "should we staff a designer?" to "here's the actual work, how do we improve it?"
- Code-based prototypes unlock edge-case exploration: zero states, startup vs. enterprise data, extreme volume—impossible to manage in Figma files
- Built-in design review mode: AI-generated comment summaries + direct-to-AI fix dispatch
- Owen's rule: "as soon as you feel like swearing, /clear and start again"—avoid sunk-cost fallacy in agentic workflows

## Why It Matters

Protodash shows that the highest-leverage internal AI tools are ones deeply wired to a company's specific design system and culture—and that building them is now within reach of a single motivated engineer-designer.
